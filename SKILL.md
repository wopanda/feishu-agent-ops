# Feishu Agent Ops

这是一个把多个飞书机器人整理并接入 OpenClaw 多 Agent 的部署编排 skill。

它优先负责：
- 新增多 Agent
- 扩容现有多 Agent
- 巡检配置健康度
- 修复常见接入异常

## 前台交互原则（更新）

默认不要一上来给用户一长串字段表单。

前台要遵守这 4 条：

1. **先识别场景，再补最少信息**
2. **能推断的字段，不先让用户填**
3. **缺信息时，要告诉用户“去哪里找”**
4. **新增/扩容先给预览，排障先做只读扫描**

## 哪些信息是“必要信息”

### 新增 / 扩容场景
默认最低只要：
- `appId`
- `appSecret`
- `botName`

### 默认可由系统推断 / 生成的信息
- `accountId`：可按 botName 自动生成稳妥命名
- `agentId`：可先扫描当前环境，再推荐绑定哪个已有 agent，或建议新建
- `roleName`
- `model`
- `isDefault`

### 只有这些场景才继续追问
- 用户明确要求“绑定已有 Agent” → 再问 / 推断 `agentId`
- 用户明确要求“群聊级绑定” → 再问 `chatId`
- 用户明确要求“设为默认 Agent” → 再确认 `isDefault`

## 必须给的“去哪找”提示

当用户缺少必要信息时，要直接告诉他去哪找：

### `appId` / `appSecret`
路径：
- 飞书开放平台 → 对应应用 → **凭证与基础信息**

### `botName`
可直接使用：
- 飞书开放平台里的应用名称
- 或飞书里显示的机器人名称

### `accountId`
不是飞书官方字段，而是 OpenClaw 本地配置里的内部代号。
如果用户不会起名，默认由系统生成，不先卡住用户。

### `agentId`
不是先让用户自己找。
更稳的方式是：
- 先扫描当前 OpenClaw 环境
- 再把可复用的 agentId 列给用户选

## 默认前台话术方向

### 新增一个机器人时
优先引导成这种最小输入：

```text
我想新增一个飞书机器人：
- appId：cli_xxx
- appSecret：xxx
- botName：日新调研
其他你帮我补。
```

### 增量扩容时
优先引导成：

```text
我已经有多机器人了，下面这个帮我增量加进去：
- appId：cli_xxx
- appSecret：xxx
- botName：内容助手
不要覆盖旧配置。
```

### 排障时
优先引导成：

```text
这套多飞书机器人有问题：
- 现象：某个 bot 不回复 / 串会话 / 路由错
- 先帮我检查，不要直接改
```

## 场景入口

### 场景 A：一只龙虾变多只（bootstrap）
适用于：
- 当前主要还是单 bot / 单 Agent
- 想拆成多个专职机器人
- 希望先看预览再落地

### 场景 B：多只龙虾继续扩更多（expand）
适用于：
- 已经有多 Agent / 多 bot
- 想继续增量扩进去
- 不想覆盖旧配置

### 场景 C：多只龙虾出了问题（diagnose）
适用于：
- 串会话
- 某个 bot 不回复
- 路由偶尔跑错
- 升级后插件链 / 字段结构漂移，担心修偏

## 渐进式披露规则

### 第 1 步：先识别场景
只先判断属于 `bootstrap / expand / diagnose` 哪一类。

### 第 2 步：只问最少问题

#### bootstrap / expand 默认只问：
- 新 bot 的基础信息是什么？（默认只要 `appId / appSecret / botName`）
- 绑定已有 Agent，还是新建 Agent？（如果用户没说，可先扫描后建议）
- 账户级绑定，还是群聊级绑定？（如果用户没提群聊，不先追问）

#### diagnose 默认只问：
- 异常现象是什么？
- 哪几个 bot 受影响？
- 是否允许先做只读扫描？

### 第 3 步：必要时先 compat-scan
如果满足这些情况，先做兼容探测：
- 不确定当前 OpenClaw 是哪个版本
- 不确定生效的是旧 `feishu` 还是官方 `openclaw-lark`
- 升级后怀疑字段结构有变化
- 当前现场存在插件覆盖、历史残留、多版本混用

### 第 4 步：再扫当前现场
兼容探测确认没走偏后，再把当前现场压成结构化 observed state。

### 第 5 步：新增/扩容先给预览
- 先构造目标预览
- 先校验预览
- 先生成 patch preview
- 用户确认后再 apply

### 第 6 步：排障先给根因判断
- 先读现场
- 先给根因与证据
- 再决定是否修

## Root-Cause-First（根因优先）规则

在 `diagnose` 场景中，必须按下面顺序排查：

### 第 1 层：会话隔离层
- 检查 `session.dmScope`
- 多账号飞书场景优先关注：`per-account-channel-peer`

### 第 2 层：账号与路由闭环层
- 对齐 `channels.feishu.accounts` 与 `bindings`
- 检查 `accountId -> agentId` 是否闭环
- 检查默认 Agent 是否抢路由

### 第 3 层：目录与 agent 结构层
- 检查 `workspace-*`
- 检查 `agents/<id>/agent`
- 检查目录与路由映射是否一致

### 第 4 层：插件与运行层
- 检查插件重复覆盖告警
- 检查 status / logs 中的运行告警

## 自动修复 vs 需确认

### 默认可自动修复
- 缺少 binding
- 缺少 `workspace` / `agentDir`
- 可确定且无歧义的映射缺口

### 必须先确认再修
- 修改 `session.dmScope`
- 调整默认 Agent
- 涉及插件层配置切换
- 可能影响已有对话连续性的改动

## 安全护栏

- 未备份，不执行 `apply`
- 未预览，不直接改配置
- 遇到命名冲突 / 默认 Agent 冲突 / binding 冲突时先停下
- `repair` 默认只做最小修复，不大范围重写
- 没有验证依据，不说“已完成可用”

## 默认输出结构

1. 已做改动
2. 验证依据
3. 风险 / 未完成项
4. 下一步建议

## 一句话心法

> 先让用户把事说出来，再只补必要信息；能我推断的，不先让用户填表。
