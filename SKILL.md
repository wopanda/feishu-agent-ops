# Feishu Agent Ops

这是一个把多个飞书机器人整理并接入 OpenClaw 多 Agent 的部署编排 skill。

它优先负责：
- 新增多 Agent
- 扩容现有多 Agent
- 巡检配置健康度
- 修复常见接入异常

它不是只解释配置，也不是直接盲改线上环境。
默认工作方式是：

1. 先识别当前 OpenClaw / Feishu 配置现状
2. 再整理用户提供的机器人信息
3. 先生成规划预览
4. 用户确认后再执行写入
5. 最后做巡检与最小验证建议

## 何时使用

以下场景优先使用本 skill：

- 用户已经创建好多个飞书机器人，并拿到了 `appId` / `appSecret` / `accountId`
- 用户想把 1 个 Agent 扩成多个
- 用户已经有多 Agent，想继续扩容
- 用户怀疑 `bindings` / `workspace` / `dmScope` / 路由有异常，想巡检或修复

## 不适用场景

以下场景不优先使用本 skill：

- 还没有创建飞书机器人，也还没拿到 `appId` / `appSecret`
- 只是想了解 OpenClaw 配置结构，不打算实际接入
- 只是一次性临时改配置，不打算沉淀成可复用流程
- 需要自动去飞书开放平台创建机器人（这不属于本 skill 的 V1 范围）

## 首次成功路径

### 你要准备什么
每个飞书机器人至少提供：
- `accountId`
- `appId`
- `appSecret`
- `botName`

### 你怎么开始
直接说：

> 我已经创建好了这批飞书机器人，帮我接成 OpenClaw 多 Agent，先给我预览，再执行。

### 你会得到什么
- 一份多 Agent 规划预览
- 建议新增的 `agents` / `accounts` / `bindings`
- 建议创建的 `workspace` / `agentDir`
- 确认后的安全落地结果
- 巡检结果与下一步测试建议

### 怎么确认第一次成功
至少满足：
- 配置成功写入
- 目录成功创建
- 巡检无致命错误
- 至少 1 个新 bot 完成一次真实回复验证

## 支持的动作

### 1. `plan`
只分析，不落地。
适用于：先看方案、先看预览、先查冲突。

### 2. `apply`
正式执行接入或扩容。
适用于：用户确认后做备份、patch、建目录、校验。

### 3. `inspect`
巡检当前多 Agent 配置。
适用于：检查 `bindings`、`workspace`、`dmScope`、目录完整性与常见异常。

### 4. `repair`
做最小修复。
适用于：某几个 bot 不回复、绑错 Agent、目录缺失、配置不一致等场景。

## 推荐触发话术

### 新增 / 扩容
- 我已经创建好了这批飞书机器人，帮我接成 OpenClaw 多 Agent，先给我预览，再执行。
- 我现在已经有几个 Agent 了，下面是新机器人的信息，帮我增量扩进去，不要覆盖旧配置。
- 帮我把 1 个 Agent 扩成 5 个角色机器人。

### 巡检
- 帮我检查一下当前这套飞书多 Agent 配置有没有问题。
- 看看 `bindings / workspace / dmScope` 有没有异常。
- 帮我做一次多 Agent 健康检查。

### 修复
- 现在有几个 bot 不回复，帮我定位并修一下。
- 有机器人串了 Agent，帮我检查路由问题。
- 某个 `agentDir` 或 `workspace` 可能没建对，帮我修复。

## 输入协议

### 用户最少应提供
每个机器人至少包含：
- `accountId`
- `appId`
- `appSecret`
- `botName`

### 可选补充
- `roleName`
- `agentId`
- `model`
- `isDefault`

### 输入形式
- 自然语言列表
- JSON
- YAML

## 安全护栏

- 未备份，不执行 `apply`
- 未预览，不直接改配置
- 遇到命名冲突、默认 Agent 冲突、binding 冲突时先停下
- `repair` 默认只做最小修复，不大范围重写
- 没有验证依据，不说“已完成可用”

## 默认输出结构

对用户汇报时，默认收口成：

1. 已做改动
2. 验证依据
3. 风险 / 未完成项
4. 下一步建议

## 一句话心法

> 不是把一批机器人“配进去”，而是把它们“安全地整理成可运行的多 Agent 系统”。
