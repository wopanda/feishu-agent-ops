# Feishu Agent Ops Architecture V3

## 一句话定位

`feishu-agent-ops` V3 的目标不是继续做一个“大而全的多 Agent 运维入口”，而是：

> 以“新增 / 扩容龙虾”为主入口，以“排查 / 修复问题”为次入口，底层由确定性代码引擎驱动，上层由 LLM 做渐进式协调与解释。

---

## 为什么要重写

旧问题不在于功能太少，而在于：
- 前台看起来很轻
- 后台依赖很重
- 新增线和排障线混在一起
- 用户以为只是“加一个机器人”，实际却掉进完整运维编排链

这会导致：
- 首次成功率不稳
- 对外分发时容易卡死
- 问题不像“脚本失败”，更像“小龙虾卡住不动”

---

## 双支线模型

## A. 新增 / 扩容线（主入口）

### 用户目标
- 新增一个飞书机器人
- 在已有多龙虾环境中继续扩容

### 前台原则
- 默认只收最少必要信息
- 默认渐进式披露
- 默认先预览，再执行
- 默认不掉入重型排障链

### 后台原则
- 由确定性流水线负责构建目标状态与 patch
- 由代码校验决定是否允许继续
- 至少 1 个新 bot 完成真实回复验证才算闭环

---

## B. 排查 / 修复线（次入口）

### 用户目标
- 某个 bot 不回复
- 会话串路由
- 多龙虾环境异常

### 前台原则
- 明确告诉用户已进入排障模式
- 默认只读扫描
- 默认不自动修
- 修复前必须确认

### 后台原则
- compat scan 优先
- 根因优先于表面修补
- repair 默认最小化
- 修后必须验证

---

## 代码与 LLM 的边界

## LLM 负责
- 场景识别
- 最少必要提问
- 自然语言转结构化请求
- 风险解释
- 结果汇报

## 代码负责
- 配置读取与结构判断
- 插件兼容探测
- 当前状态扫描
- desired state 构造
- patch 生成
- 校验
- 执行
- 验证
- allowFrom / pairing / logs 探针

## 边界原则

不能把以下动作交给 LLM 自由决定：
- 最终 patch 内容
- 最终 apply 是否安全
- 扫描失败后的自由补洞
- 环境漂移时的强行 patch

LLM 可以提建议，但落配置必须经过代码校验。

---

## 内部流水线

## 新增 / 扩容线

```text
normalize_request
-> scan_current_state
-> build_desired_state
-> validate_plan
-> generate_patch
-> verify_setup
```

## 排障 / 修复线

```text
scan_openclaw_compat
-> scan_current_state / runtime verify
-> root-cause
-> minimal repair plan
-> verify
```

## 对外发布门槛

重新上传 GitHub / Gitee 前，至少满足：
- 双支线文档已完成分流
- README 首页主打新增 / 扩容
- 排障线明确为次入口
- 文档承诺与当前脚本能力基本一致
- 至少走通一次新增预览校核
- 至少走通一次排障验证校核

---

## 当前 V3 的核心判断

V3 不是“继续把能力往一个大 skill 里堆”，而是：

> 把现有能力改造成一个更稳的产品入口：
> 前台简单，后台确定；
> 主线顺滑，支线明确；
> LLM 协调，代码兜底。
