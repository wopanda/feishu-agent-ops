# Feishu Agent Ops

把飞书机器人接成 OpenClaw 多 Agent。

只做两件事：
- 添加机器人
- 排查问题

目标很简单：**少问、少确认、少打扰，做完主动给结果。**

---

## 最小输入

### 添加机器人
直接给：
- `appId`
- `appSecret`
- `botName`

示例：

```text
我想添加一个飞书机器人：
- appId：cli_xxx
- appSecret：xxx
- botName：日新调研
其他你帮我补。
```

### 排查问题
直接说现象。

示例：

```text
这套多飞书机器人有问题，你先检查。
```

---

## 默认行为

### 添加机器人
系统默认会：
1. 先扫描当前环境
2. 自动补全能推断的信息
3. 给最小预览
4. 真正写配置前再确认一次
5. 完成后主动告诉你结果

### 排查问题
系统默认会：
1. 直接开始只读扫描
2. 给根因判断
3. 只在必要时给修复预览
4. 完成后主动告诉你结果

---

## 默认不追问的东西

如果没必要，默认不先问：
- `accountId`
- `agentId`
- `chatId`
- `roleName`
- `model`
- `isDefault`

规则：
- `accountId` 不会填就自动建议
- 只有明确要绑定已有 Agent，才问 `agentId`
- 只有明确要群聊级绑定，才问 `chatId`

---

## 只在这些情况确认

只有以下动作才需要确认：
- 真正写配置并执行 apply
- 修改 `session.dmScope`
- 调整默认 Agent
- 插件层切换
- 可能影响已有会话连续性的改动

读扫描、生成预览、自动补命名，默认不确认。

---

## 输出风格

默认最多 3 段：
1. 结论
2. 关键依据
3. 必要时的一个下一步

执行过程中只允许非常短的状态提示，例如：
- 我先检查
- 我先处理，完成后直接告诉你结果

不承诺具体等待时间。

---

## 完成反馈

一旦完成 `apply / repair / verify` 并拿到验证依据，必须主动回复：
- 已完成，或
- 已发现问题

不能让用户靠等待去猜。

---

## 一句话原则

> 用户只管把需求说清楚；系统先判断、先扫描、先给结果。少问，少确认，少打扰。
